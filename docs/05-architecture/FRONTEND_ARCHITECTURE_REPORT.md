# PedagogyX: Frontend Architecture & Strategy Report

**Status:** Draft v1.0
**Owner:** Senior Frontend Developer & UI Systems Architect
**Date:** 2026-05-26
**Scope:** PedagogyX Web Admin Dashboard & Frontend Ecosystem (Phase 1)

As the Senior Frontend Developer & UI Systems Architect, this report establishes the long-term frontend architecture, interaction strategy, and engineering standards for the PedagogyX ecosystem. This architecture focuses specifically on the Web Admin Dashboard (Supervision Mode) utilized by school principals and deans.

---

## User & Product Analysis

**User Workflows:**

1. **Live Observation (Hot Path):** Principals monitor ongoing classes through the dashboard. They need immediate visibility into M-A (Coverage) and real-time, rolling talk-ratio estimates (PRELIMINARY).
2. **Post-Lesson Review (Cold Path):** Administrators review authoritative Pedagogy Index scores generated by the batch pipeline, scrub synchronized video/audio, and prepare coaching feedback.
3. **Intervention Triggering (M-C):** Administrators triage "Flagged" lessons in the queue that require review within 48 hours.

**Usability Considerations:**

- Administrators (Principals) are frequently interrupted, time-poor, and operate in high-cognitive-load environments. The UI must aggressively prioritize M-A and M-B metrics above the fold.
- "Preliminary" live scores must be visually distinct from "Authoritative" batch scores to prevent premature punitive actions.
- The interface must reduce surveillance anxiety by presenting data through a "coaching" and "professional development" lens.

**Edge Cases:**

- Disconnected or failing stream ingest (e.g., Android DAT client loses connection). The UI must gracefully degrade and show clear, non-technical error states.
- Missing authoritative scores due to pipeline failures or prolonged inference queuing on the OSS central servers.

**Accessibility Considerations:**

- Dashboards must be legible on sub-optimal displays (low brightness, high glare) and projectors.
- Interfaces require high contrast, tabular data presentation, and dual-language support (Hindi/English) for the Indian market.

## Frontend Architecture

**Component Structure:**

- **Atomic Design Principles:** We will structure the Next.js `app` router application utilizing a strictly typed component library (built with Radix UI primitives and Tailwind CSS).
- Components will be split into `ui` (dumb, presentational components) and `features` (smart components mapped to domain logic, e.g., `LessonScrubber`, `PedagogyScorecard`).

**State Management:**

- Server-side state and data fetching will be handled natively via Next.js App Router (React Server Components) and `fetch` with aggressive `revalidate` caching strategies.
- Client-side state (UI toggles, video playback state, scrubber position) will utilize lightweight context or Zustand to prevent unnecessary prop-drilling.

**Routing:**

- **Next.js 15 App Router:** leveraging nested layouts to maintain persistent navigation (Sidebars, Global Headers) while deep-diving into specific `[lessonId]` routes.

**Design System Strategy:**

- Establish a single source of truth for design tokens (colors, typography, spacing) mapped via Tailwind CSS configuration.
- Implement a structured "Status" token system (`status-preliminary`, `status-authoritative`, `status-flagged`) to enforce visual consistency across all metrics.

## UI/UX Strategy

**Interaction Design:**

- **Zero-Friction Scrubber:** The most complex UI element will be the video player/timeline scrubber. It must synchronize video playback with a timeline of pedagogical events (e.g., high teacher talk time).
- Hover states and focus rings will be over-indexed to guarantee discoverability for non-technical users.

**Responsiveness:**

- The Web Admin Dashboard will adopt a responsive bento-box grid for the home overview, naturally collapsing to a single column on tablet views (principals walking the halls).

**Accessibility:**

- Focus transitions must follow the visual reading order. Semantic HTML5 tags (`<article>`, `<section>`, `<nav>`) will map to ARIA roles.
- Text scaling must not break the data tables; we will employ horizontal scroll boundaries for dense tabular data rather than text truncation.

**Visual Hierarchy:**

- M-A (Observation Coverage) is the primary CTA and will command the highest visual weight.
- Color will be used sparingly; red/green will not be used as the sole indicator for status. Icons and typography weight will act as secondary indicators.

**Animation Systems:**

- Animations will be purposeful and minimal. Expanding/collapsing accordion rows for "Flagged" lessons will use smooth 200ms ease-in-out transitions to reduce cognitive jarring.

## Performance Optimization

**Rendering Optimization:**

- Maximize the use of React Server Components (RSC) to ship zero JS for the initial dashboard shell. Client components (`"use client"`) will be strictly isolated to interactive leaves (e.g., the video player).

**Lazy Loading:**

- Heavy modules, such as video player libraries or complex charts (e.g., Recharts for longitudinal growth), will be dynamically imported.

**Bundle Optimization:**

- We will routinely audit bundle size using `@next/bundle-analyzer`.
- Tree-shaking will be enforced for all third-party dependencies.

**Caching Strategy:**

- Static assets and initial JSON payloads will be cached at the edge.
- Live data (Hot Path) will utilize lightweight polling or Server-Sent Events (SSE) rather than heavy WebSockets, keeping the network tab clean and reducing server overhead.

---

## Accessibility Strategy

**Semantic Structure:**

- The DOM must make sense with CSS disabled. Strict adherence to heading hierarchies (`h1` -> `h2` -> `h3`).

**Keyboard Navigation:**

- All interactive elements must be reachable via the `Tab` key. Custom components (like the timeline scrubber) must support arrow key navigation for precise seeking.

**ARIA Strategy:**

- `aria-live="polite"` will be applied to the Hot Path live metrics to notify screen readers of updates without interrupting the user.
- Explicit `aria-labels` on all icon-only buttons (e.g., Play, Pause, Export PDF).

**Responsive Accessibility:**

- Ensure touch targets on tablet views meet the minimum 48x48dp requirement to accommodate principal tablet usage.

---

## Testing Strategy

**Component Testing:**

- Use Vitest and React Testing Library for isolated unit tests on critical UI primitives (e.g., formatting functions, accessible buttons).

**Interaction Testing:**

- Verify state transitions in complex components (e.g., ensuring the video scrubber updates state correctly when dragged) using React Testing Library.

**Accessibility Testing:**

- Integrate `axe-core` into the CI pipeline and use `jest-axe` within unit tests to catch contrast and ARIA violations automatically.

**Regression Testing:**

- Visual regression testing (e.g., Playwright) for the core dashboard layouts to ensure CSS refactoring does not break the bento-box grid or table alignments.

## Frontend DevOps

**CI Workflows:**

- GitHub Actions pipeline (`.github/workflows/test.yml`) will mandate `next lint`, `vitest` unit tests, and Prettier formatting on every PR.

**Deployment Optimization:**

- Given the self-hosted OSS requirement, the Next.js app will be packaged as a standalone Docker container (`output: 'standalone'`), minimizing the image size and cold start time.

**Frontend Observability:**

- Implement structured client-side logging. Unhandled promise rejections and React Error Boundaries will post structured JSON to the backend observability stack (OTel).

**Monitoring:**

- Track Core Web Vitals (LCP, CLS, INP) specifically on the K-12 network conditions (simulated 3G/slow 4G) to ensure the interface remains snappy in the target environment.

## Refactoring Opportunities

**Simplifications:**

- The current MVP boilerplate `services/web/app/page.tsx` uses raw inline styles (`style={{ display: "flex" }}`). This must be entirely replaced with a utility-first CSS framework (Tailwind CSS) to ensure maintainability and consistency.

**Modularization:**

- Extract the monolithic API fetching logic in `page.tsx` into a dedicated `lib/api/` boundary.
- Deconstruct the `HomePage` component into isolated feature components (`SchoolOverview`, `MetricsCard`, `RecentSessionsTable`).

**Maintainability Improvements:**

- Enforce strict TypeScript interfaces for all API payloads (e.g., `TeacherTalkRatio`, `PedagogyIndex`) to prevent runtime `undefined` errors.
- Integrate an ESLint rule to forbid inline styles.

## Risks & Tradeoffs

**Browser Limitations:**

- **Risk:** Older tablet browsers in Indian schools may not fully support modern CSS grid/subgrid features.
- **Tradeoff:** Use Flexbox fallbacks where appropriate, though we will establish a baseline browser support matrix (e.g., modern Chromium/Safari only) to avoid excessive polyfilling overhead.

**Scalability Concerns:**

- **Risk:** High frequency of Hot Path updates overwhelming the React render cycle on low-end admin machines.
- **Tradeoff:** Debounce live metric updates to a maximum of 1 update per 5 seconds, rather than attempting true real-time streaming.

**Performance Tradeoffs:**

- **Risk:** Including large localization dictionaries (Hindi/English) in the main bundle.
- **Tradeoff:** Implement dynamic loading for i18n dictionaries to keep the initial English load as fast as possible.
