# Product Design & Human Experience Architecture Report

**Status:** Draft v1.0
**Date:** 2026-05-20
**Role:** Autonomous Elite Product Design & Human Experience Systems Architect

## 1. User & Experience Analysis

PedagogyX serves two primary personas with conflicting psychological needs.

- **The Teacher:** Experiences the system as a potential threat (surveillance) but needs it to be a trusted developmental tool. They are time-poor and highly sensitive to unfair criticism by an AI.
- **The Administrator (Principal/Dean):** Needs bird's-eye visibility into instructional quality to enforce standards and identify struggling teachers, but must avoid drowning in data.

## 2. UX Strategy

- **Trust through Transparency:** The system must never present an AI conclusion as absolute fact. Every generated insight ("Low Engagement", "Poor Wait Time") must be directly linked to a clickable video timestamp.
- **Progressive Disclosure:** Dashboards should show high-level health scores first. Users must actively click to reveal granular metrics, preventing cognitive overload.
- **Empathetic Tone:** Coaching text generated by the LLM must be explicitly prompted to use supportive, non-punitive language, adhering to regional pedagogical norms.

## 3. UI Strategy

- **Minimalism:** The interface must not compete with the video content. We utilize a muted, neutral color palette (grays, soft blues) reserved entirely for data visualization and critical alerts.
- **Data Density:** The Admin Dashboard requires high data density (table views, sparklines) to allow scanning of 50+ classrooms. The Teacher Portal requires low density, focusing deeply on a single lesson.

## 4. Interaction Design

- **The Multimodal Scrubber:** The core interaction paradigm for teachers is an advanced video scrubber. Below the video timeline, discrete lanes visualize different data streams: a lane for speaker diarization, a lane for engagement heatmap, and a lane for AI bookmarks.
- **Frictionless Capture:** The capture agent (Windows/Android) must be a "one-click" operation. It should automatically detect default cameras and mics, requiring zero configuration from a non-technical teacher.

## 5. Information Architecture

- **Hierarchy:**
  1.  Organization (District)
  2.  Location (School)
  3.  User (Teacher)
  4.  Asset (Lesson Session)
- **Navigation:** A persistent sidebar allows quick switching between "Live Supervision," "Historical Reports," and "My Portfolio" (for teachers).

## 6. Accessibility Strategy

- **Color Independence:** Never rely solely on color to convey meaning (e.g., a red dot for "bad"). Always accompany color with an icon or text label.
- **Keyboard Navigation:** The video scrubber and complex data tables must be fully navigable via keyboard, essential for users with motor impairments.
- **Screen Readers:** All AI-generated graphs must have descriptive `aria-labels` summarizing the trend (e.g., "Line graph showing engagement dipping during the last 10 minutes").

## 7. Design System Strategy

- **Component Library:** Build atop shadcn/ui and Tailwind CSS to ensure rapid, consistent development.
- **Theming:** Implement a robust theming engine from day one to support "Supervision Mode" (serious, data-heavy) vs. "Coaching Mode" (lighter, encouraging) styling.

## 8. UX Research Plan

- **Phase 1 (Pre-MVP):** Conduct 5 contextual inquiry sessions with Indian school principals to understand their daily monitoring workflows and acceptable thresholds for "supervision."
- **Phase 2 (Pilot):** Diary studies with 10 teachers using the platform for two weeks to measure trust and anxiety levels related to the AI feedback.

## 9. Risks & Tradeoffs

- **Risk:** "Dashboard Fatigue." Providing too much analytics makes the system useless. **Tradeoff:** We will intentionally hide complex metrics (like precise acoustic data) from the default view, prioritizing the synthesized Pedagogy Index, even if it feels "dumbed down" initially.
- **Risk:** Surveillance Backlash. **Mitigation:** The UX must explicitly highlight privacy controls and data retention policies within the teacher's view, reinforcing their agency.

## 10. Agile Sprint Plan

- **Sprint 1:** Finalize high-fidelity wireframes for the Admin Supervision Dashboard and the Teacher Video Review interface in Figma.
- **Sprint 2:** Develop the core Design System components in React (Buttons, Data Tables, Charts) using Storybook.
- **Sprint 3:** Build the interactive prototype of the Multimodal Scrubber component, linking dummy JSON data to visual timeline tracks.
