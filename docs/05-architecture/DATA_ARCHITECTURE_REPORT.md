# Data Architecture Report

**Status:** Draft v1.0
**Date:** 2026-05-20
**Role:** Autonomous Senior Data Engineer

## 1. Data Problem Analysis

The platform generates massive, diverse datasets: raw multi-stream video/audio, high-velocity time-series inference events (e.g., student pose every second), structured metadata (teachers, schools, rubrics), and unstructured semantic text (transcripts, coaching narratives). Operating under the strict DPDP privacy laws requires absolute certainty in data lifecycle management, tenant isolation, and the ability to process unstructured data efficiently at scale.

## 2. Data Architecture

A modern, decoupled data stack optimized for both ML inference and rapid analytics.

- **Operational DB:** PostgreSQL (metadata, state, RBAC).
- **Event/Analytics DB:** ClickHouse (high-velocity telemetry, engagement scores).
- **Object Store:** MinIO (raw media blobs, compiled ML artifacts).
- **Vector Store:** Qdrant (embeddings for RAG).

## 3. Pipeline Design

- **Ingest Pipeline:** Capture Agent -> Edge Node (LAN buffer) -> Cloud Ingest (MediaMTX) -> MinIO. This pipeline guarantees fault tolerance over bad WAN links.
- **Inference Pipeline:** MinIO (Trigger) -> Celery Task Queue -> GPU Workers. Results (JSON artifacts) are written back to MinIO.
- **Analytics Aggregation Pipeline:** A scheduled process reads the raw JSON artifacts, transforms them into structured relational tables (for Postgres dashboards) and wide columnar tables (for ClickHouse heatmaps).

## 4. Storage & Warehouse Design

- **MinIO:** Architected with strict bucket policies. `raw-media` has aggressive lifecycle policies (delete after 30 days post-inference). `ml-artifacts` holds anonymized JSON transcripts indefinitely.
- **ClickHouse:** Acts as the primary analytics warehouse. Tables are optimized using `ReplacingMergeTree` to handle deduplication from worker retries.

## 5. Data Quality & Governance

- **Schemas:** Pydantic models define strict contracts for all JSON artifacts generated by ML pipelines.
- **Validation:** Great Expectations (or custom equivalent) runs daily to ensure the aggregations in ClickHouse match the raw artifacts in MinIO (e.g., preventing silent data loss in the queue).
- **Governance:** Under DPDP, a clear map of PII vs non-PII is maintained. Video is PII; aggregated bounding box coordinates are not.

## 6. Observability

- **Data Lineage:** Every row in ClickHouse and every coaching summary in Postgres contains a `trace_id` linking it directly to the exact Celery task run and the source MinIO blob.
- **Pipeline Metrics:** Prometheus alerts on "time-since-last-successful-batch" to detect silent pipeline failures.

## 7. Security & Compliance

- **DPDP:** Implemented cascading deletes. If a parent revokes consent, a script traverses MinIO and Qdrant to purge associated data without requiring a full system wipe.
- **Encryption:** Customer Managed Keys (CMK) via AWS KMS (or Vault if on-prem) encrypts all MinIO buckets and Postgres volumes.
- **Tenant Isolation:** Row-Level Security (RLS) is heavily utilized in PostgreSQL to ensure no cross-school data leakage.

## 8. Performance Optimization

- **ClickHouse Materialized Views:** Used aggressively to pre-calculate the "Pedagogy Index" rollups for the Admin Dashboard, reducing dashboard load times from seconds to milliseconds.
- **Postgres Indexing:** GIN and GiST indexes applied to JSONB columns where unstructured querying is necessary before migration to ClickHouse.

## 9. Risks & Tradeoffs

- **Risk:** Managing multiple distinct databases (Postgres, ClickHouse, Qdrant, MinIO) increases operational overhead for a small team. **Tradeoff:** Using Postgres for everything (via extensions like Timescale and pgvector) would simplify ops but drastically bottleneck the ML ingestion rate and dashboard performance at scale. We accept the complexity for required performance.

## 10. Agile Sprint Plan

- **Sprint 1:** Define strict Pydantic schemas for all ML artifacts. Provision Postgres and MinIO. Implement RLS in Postgres.
- **Sprint 2:** Provision ClickHouse. Write the Aggregation Pipeline that pulls dummy ML JSON files from MinIO and populates ClickHouse.
- **Sprint 3:** Implement the DPDP Compliance tooling (cascading delete scripts, automated data retention sweepers for MinIO).
