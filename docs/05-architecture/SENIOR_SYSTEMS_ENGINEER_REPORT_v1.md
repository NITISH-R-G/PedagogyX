# PedagogyX Autonomous Systems Engineering Report

## System Overview

**Purpose:** PedagogyX is an educational analytics tool utilizing Meta Ray-Ban smart glasses to capture edge data, serving real-time analytics to a web client. The system acts as the primary pipeline for processing edge-captured multimodal data (video, audio, metrics), requiring world-class ingestion, processing, storage, and retrieval mechanisms.

**Requirements:**

- High-throughput multimodal data ingestion from Meta Ray-Ban (via Android companion app).
- Low-latency real-time video/audio processing (CV/ASR workers).
- Resilient API backend for servicing frontend analytics queries.
- Modular deployment enabling scaling up of workers decoupled from web traffic.
- Zero-downtime rolling updates.

**Scale Assumptions:**

- Target 10,000 active concurrent pilot devices.
- Payload streams up to 100s MB/min per device (burst mode).
- 10M API requests per day.
- Massive data storage over time (10s of TBs over 6 months).
- 99.99% Availability expectation.

**Constraints:**

- G2 Data Regulations: PII protection, blocking real pilot data initially, strictly bound to synthetic metrics until clearance.
- Hardware constraints: Current CPU-only baseline, GPU expansion required for scaling processing workloads (ASR/CV) post MVP.

## High Level Architecture

**Major Components:**

- **Edge Client:** Meta Ray-Ban Smart Glasses -> Android Capture Companion App (Wearables Device Access Toolkit).
- **API Gateway / Ingress:** Global load balancer terminating TLS, routing traffic.
- **Core API (`services/api`):** FastAPI based REST backend. Handles authentication, routing, and metadata persistence.
- **Web Client (`services/web`):** Next.js 15, Tailwind CSS v4 frontend.
- **Processing Workers (`services/worker-cv`, `worker-asr`, `worker-metrics`):** Asynchronous python workers pulling off message queues.
- **Message Broker:** Kafka or RabbitMQ for decoupling ingestion from processing.
- **Object Store:** S3-compatible blob storage for raw video/audio.
- **Database:** PostgreSQL for structured metadata, session logs, user profiles.
- **Cache:** Redis for session state, rate limiting, and frequent query caching.

**Interactions & Data Flow:**

1. Smart glasses capture video/audio -> Android Companion syncs.
2. Android Companion pushes raw payloads to Object Store (via presigned URLs generated by Core API).
3. Android Companion sends sync metadata to Core API.
4. Core API drops a processing job into the Message Broker.
5. `worker-cv` / `worker-asr` consume jobs, read from Object Store, run ML pipelines, write processed results to Database.
6. Web Client queries Core API (cached via Redis) to visualize processed analytics.

## Infrastructure Design

**Deployment Topology:**

- Kubernetes (K8s) cluster as the primary orchestration plane.
- Multi-Availability Zone (AZ) deployment for high availability.
- Web, API, and Worker components deployed as distinct K8s Deployments with Horizontal Pod Autoscaling (HPA).

**Cloud Architecture:**

- Managed Kubernetes Service (EKS/GKE/AKS).
- Managed PostgreSQL (Aurora/Cloud SQL) for durability and automated backups.
- Managed Redis (ElastiCache/MemoryStore).
- Managed Message Broker (MSK/PubSub) to reduce operational overhead.
- Cloudflare or AWS CloudFront for CDN and WAF.

**Networking:**

- Virtual Private Cloud (VPC) with public/private subnets.
- Ingress controllers (NGINX/ALB) in public subnets.
- Compute (EKS nodes) and Databases strictly in private subnets.
- NAT Gateways for outbound internet access from workers.

## Database Design

**Schema Strategy:**

- PostgreSQL relational model prioritizing normalization for metadata.
- JSONB columns utilized for flexible, schema-less worker output payloads.
- **Core Tables:** `users`, `devices`, `sessions`, `media_assets`, `analysis_results`.

**Replication & Scaling:**

- Primary-Replica topology: Single Primary for writes, multiple Read Replicas.
- Core API configured with read/write connection pools, directing heavy analytic `SELECT` queries to Read Replicas.

**Consistency Model:**

- Strong consistency for critical flows (user creation, device registration).
- Eventual consistency for analytics (workers process asynchronously, web client sees data when DB is updated).

## Scalability Strategy

**Horizontal Scaling:**

- Stateless API and Web services scaled automatically via K8s HPA based on CPU/Memory usage.
- Worker services scaled based on Message Broker queue depth (e.g., KEDA).

**Caching:**

- Redis for heavily requested analytical aggregates and session state.
- CDN caching for static frontend assets (Next.js build).

**Partitioning & Load Balancing:**

- API Gateway routes traffic across K8s nodes.
- Database partitioning by time (e.g., `sessions` partitioned by month) to maintain query performance as data grows.

## Reliability Strategy

**Failover & Redundancy:**

- Multi-AZ deployment across all compute and managed services.
- Database automated failover with sub-minute recovery target.

**Resilience Patterns:**

- **Circuit Breakers:** Implemented in Core API around downstream dependencies (e.g., Object Store presigned URL generation) to prevent cascading failures.
- **Retries with Exponential Backoff:** Worker processes retrying failed S3 reads or DB writes.
- **Dead Letter Queues (DLQ):** Failed processing jobs routed to DLQs for manual inspection and replay.

## Security Architecture

**Authentication & Authorization:**

- OAuth2/OIDC for Web Client authentication.
- API Key / mTLS authentication for Edge Client (Android App) to Core API.
- Role-Based Access Control (RBAC) ensuring users only see their organization's data.

**Encryption:**

- TLS 1.3 for all data in transit.
- AES-256 encryption at rest for S3 buckets and Database volumes.
- Secrets management via HashiCorp Vault or AWS Secrets Manager (no hardcoded credentials).

**Abuse Prevention:**

- WAF (Web Application Firewall) blocking common OWASP top 10 vulnerabilities.
- API Gateway level rate limiting (Token Bucket algorithm) per device/user to prevent DDoS and noisy neighbor problems.

## Observability Stack

**Logging:**

- Structured JSON logging across all microservices.
- FluentBit daemonsets forwarding logs to centralized logging platform (Elasticsearch/Datadog/CloudWatch).

**Tracing:**

- OpenTelemetry distributed tracing passing trace contexts from Web Client -> API -> Message Broker -> Workers.
- Allows pinpointing latency bottlenecks in complex asynchronous workflows.

**Monitoring & Alerting:**

- Prometheus scraping metrics (CPU, Memory, HTTP status codes, queue depth).
- Grafana dashboards for visualizing system health.
- PagerDuty integration for critical alerts (e.g., API error rate > 1%, DLQ depth > 100).

## Performance Optimization

**Bottlenecks Addressed:**

- **I/O Bound Workers:** Python workers utilize async/await patterns heavily to maximize throughput while waiting on network/disk.
- **Heavy Analytics Queries:** Addressed via DB Read Replicas and Redis caching.

**Latency Optimization:**

- Edge CDN terminating TLS closer to the user.
- Direct-to-S3 uploads for media assets (bypassing Core API) reduces network hops and API load.

**Throughput Optimization:**

- Batching database inserts in worker processes.
- Connection pooling (pgbouncer or native async pools) to prevent connection starvation under load.

## Tradeoffs

**Pros:**

- Highly decoupled architecture enables independent scaling of compute-heavy (workers) vs I/O-heavy (API) workloads.
- Utilization of managed services reduces operational burden.

**Cons:**

- Increased operational complexity due to microservices and asynchronous message queues.
- Eventual consistency of analytics requires careful UI/UX design (e.g., showing "Processing..." states).

**Limitations & Future Improvements:**

- Current lack of GPU workers limits CV/ASR processing speed. Future sprints must integrate GPU node pools for workers.
- Introduce real-time streaming (WebSockets) for live feedback to the Android App rather than polling.

## Agile Sprint Plan

### Sprint 1: Core Foundation & Ingestion

- Setup Infrastructure as Code (Terraform) for VPC, EKS, RDS, S3.
- Deploy Core API and Web MVP.
- Implement Edge-to-S3 direct upload flow with presigned URLs.

### Sprint 2: Asynchronous Processing Pipeline

- Deploy Message Broker (RabbitMQ/MSK).
- Implement basic `worker-cv` and `worker-asr` structure.
- Wire API to publish events to Broker; Workers to consume and update DB.

### Sprint 3: Observability & Resilience

- Deploy Prometheus, Grafana, OpenTelemetry stack.
- Implement Circuit Breakers, Retries, and DLQs across API and Workers.
- Conduct load testing and tune K8s HPA and KEDA scaling rules.

### Sprint 4: Security & Compliance (G2 Prep)

- Implement strict RBAC and Data Partitioning.
- Perform security audits (WAF, Secrets Management).
- Finalize production readiness review.
